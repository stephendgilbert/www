  • To learn the different categories of memory
  • To learn to write and use constructors and destructors
  • To be able to create classes that manage their own memory allocation and deallocation
  • To understand how to avoid common memory management errors

Categories of Memory

Memory is divided into four categories:

  1. Code - instructions for all global and member functions
  2. Static Data - global variables, and all local and member variables declared static
  3. Run-time Stack - automatics (local non-static)
  4. Free Store (Heap) - dynamically allocated (new)

Code Memory

  • Contains all machine instructions for functions and methods
  • May be referenced through pointers
  • Function pointers discussed in Section 7.6

Static Data Memory

Globals, static locals and static member variables

  • Only one value attached to a name
  • Size of statics can be known prior to execution
  • Created only once, before main()
  • Initialized only once
  • Exists until the end of execution
  • These 2 functions do the same thing: int counter = 0; int counting_fun() { // Increments at each call counter++; return counter; } int counting_fun() { static int counter = 0; // Will be initialized only once counter++; return counter; }

The Run-time Stack

  • Used for functions
  • Every call is a new block of memory, or activation record, on stack
  • Activation record contains:
    • Parameters
    • Return pointer
    • Local variables
    • Other machine-specific stuff
  • Block for current function on top of stack
  • Return from function decrements stack pointer
  • Drawbacks:
    • Local variables cease to exist when function exits
    • Size of locals must be known at compile time

The Run-time Stack

  • Setup: f invokes g, which, in turn, invokes h
  • h is on top of the stack
  • g is next on the stack
  • h must terminate before g can continue
  • Same function may be on the stack multiple times (recursion)
  • Same variable name is bound to several different values

Heap Memory

The heap, or free store:

  • Neither the number of values, nor the size of a particular value need be known at compile time
  • Lifetime of a value is not tied to a function
  • Values may persist after program ends

Dynamic memory allocation

  • A memory allocator (new) finds storage on the heap, returns a pointer
  • Values accessed through a pointer
    • Pointer itself might be on the heap, or on the stack
  • To notify the memory allocator that memory can be returned to heap: delete boss;

Common Memory Errors

  • The programmer is responsible for memory management in C++
  • Pointers can refer to memory in any of the four areas (above)
  • Possible errors include:
    • Using a value that has not been initialized
    • Using a pointer to reference a memory location that is no longer valid (dangling pointer)
    • Forgetting to delete a dynamically allocated section of memory (memory leak)
    • Deleting a memory value that was never allocated
    • Deleting a dynamically allocated section of memory more than once

Initialization Errors

  • Can occur for variables in any section
  • Use of constructors helps
  • Pointers compound this problem
  • Pointer might be uninitialized: Employee* boss; cout << "My boss earns " << boss->salary(); // Error—almost certain crash, because pointer is undefined If you're lucky, this will crash
  • Or, a valid pointer might point to uninitialized memory int* coins = new int[3]; // Array for counts of pennies, nickels, and dimes // Error.coins uninitialized double total = coins[0] + coins[1] * 0.05 + coins[2] * 0.10; The pointer is good, but the array is not initialized

Lifetime Errors

  • When a function exits, local variables are no longer valid
  • Referring to a local variable after the function exits is a subtle error
    • Until overwritten (new function call), the value appears to be there
    // Error.this function returns a reference to an invalid memory location char* read_a_line() { char buffer[200]; // Declare a buffer to hold the text cin >> buffer; // Read a line of text return buffer; // Return the text } ... char* p = read_a_line();

Figure 4 Creation of a Dangling Pointer

Array Bounds Errors

  • Assumptions about the size of statically-allocated arrays are dangerous: char buffer[200]; // Declare a buffer to hold the text
    • A line of input will not exceed 200 characters
    • Really?
  • Array bounds are not checked by the compiler
  • Any use of arrays (reading and writing) must by carefully guarded
  • Use smarter containers (vectors)

Object Slicing

  • A pointer to an object might be pointing at an object of a child class
  • The behavior of pointers and nonpointers is subtly different: TravelClock* p = new TravelClock(true, "Rome", 9); Clock* cp = p; Clock c = *p; cout << cp->get_location() << " time is " << cp->get_hours() << "\n"; cout << c.get_location() << " time is " << c.get_hours() << "\n";

The output is: Rome time is 6 Local time is 21

Memory Leaks

  • Failure to return memory
  • Not fatal, but will choke a computer over time
  • Often the result of successive assignments: Clock* a_clock; ... a_clock = new TravelClock(true, "Rome", 9); ... a_clock = new TravelClock(true, "Tokyo", -7); // Leak, old memory is now lost
  • Older objects need to be returned to the heap before a subsequent assignment

Using Invalid Memory References

  • Multiple references to a heap object will always cause problems
  • When the object is freed through one pointer, the other pointers become invalid
  • Reading from these locations might yield erroneous results
  • Writing to the locations might well foul up the memory manager
  • If that location is re-allocated, you'll foul up somebody else
  • This insidious error is extremely common: for (Node* p = ptr; p != NULL; p = p->next) { delete p; // Error—p->next is referenced after p is deleted }

    It appears to work. But, for a long list, it is probable that your program is sliced out; that memory might well be reallocated.

Deleting a Value More Than Once

  • Calling delete twice on the same location might corrupt the heap
  • Always set a pointer to NULL after call delete
  • Multiply-referenced values compound this problem: Clock* my_clock = new TravelClock(true, "Rome", 9); Clock* your_clock = my_clock; ... delete my_clock; my_clock = NULL; delete your_clock; // Same value being deleted twice
  • No easy way to avoid this
  • Be diligent

Deleting a Value That Was Never Allocated

  • Calling delete on a pointer that was never initialized will cause trouble
  • Deleting the NULL pointer is always safe: TravelClock* tc = NULL; ... delete tc; // This time OK, delete does nothing

Constructors

  • Same name as the class
  • No return type
  • Used to initialize class' attributes
  • May be overloaded
  • Only one constructor is invoked at creation
  • Implicitly invoked:
    • Each time a block is entered where an object is declared
    • For static objects - before main starts
    • When an object is created on the heap with new
    • When an object is passed by value
    • For member objects, when aggregate is created
    • When child is instantiated

Constructors

  • All constructors perform necessary initialization
  • Constructors tie together memory allocation and object initialization
  • Ensure that every value created is properly initialized
  • Encapsulate the management of the internal state of an object, along with destructors
  • Consider the string class:
    • Contents not stored in the string object
    • Stores a pointer to a character array on the heap
    • String object manages the memory itself
    • Constructors make sure pointers and memory are properly initialized
    • Destructor deallocates heap memory

Constructors - String Example

class String { public: String(); // Default constructor String(const char p[]); // Simple constructor String(const String& right); // Copy constructor ~String(); // Destructor String& operator=(const String& right); // Assignment operator String& operator+=(const String& right); int length() const; char& operator[](int index); char operator[](int index) const; private: char* buffer; int len; };

Constructors with Arguments

  • Example: Our constructor must allocate space from heap and initialize: String::String(const char p[]) { // Determine number of characters in string (strlen(p)) len = 0; while (p[len] != '\0') len++; // Allocate buffer array, remember to make space for NULL character buffer = new char[len + 1]; // Copy new characters (strcpy( buffer, p )) for (int i = 0; i < len; i++) buffer[i] = p[i]; buffer[len] ='\0'; }
  • Probably uses strlen and strcpy

Common Error

Forgetting the Dual Use of Single Argument Constructors

  • Used when explicitly creating a new object
  • Also used to convert one type into another

Consider a constructor that converts an integer into a string representation

  • Consider passing a string argument: print_name( 'a' );
  • 'a' is a number; ASCII 97
  • The resulting string is the "97" (not "a")

Use explicit keyword to inhibit implicit type conversion

Default Constructors

  • The constructor that takes no arguments
  • Invoked without parenthesis, (except for unnamed temporaries
  • Invoked:
    • When a variable is declared (or created from the heap) with no arguments
    • The only initializer for elements in an array
    • When object is member data of another class, and no initializer is provided
  • Example: a String is, by default, an empty string String::String() { len = 0; buffer = NULL; // No need to allocate array to hold zero characters }

Common Error

Default Constructor and Parentheses

  • Omit parentheses to explicitly call the default constructor: String new_name;
  • Using parentheses is syntactically correct, but is actually a function declaration: String new_name();
  • We use parentheses to create an unnamed object using the default constructor: print_time(Time()); // Pass current time to print_time

Copy Constructors

  • Used internally to make copies, or clones, of objects
  • Invoked:
    • Explicitly String first("Fred"); String second(first); // second is initialized from first using copy constructor String third = first; // Also uses copy constructor
    • During initialization using = String third = first; // Also uses copy constructor
    • When an object is passed or returned by value
  • Note: Use const when passing a reference to avoid copy overhead

Copy Constructors

  • Automatically supplied by compiler — memberwise (shallow) copy by default
    • Calls copy constructor of all member data
  • Example: each String has its own dynamically allocated buffer, so default behavior is insufficient. We extend this behavior: String::String(const String& right) { len = right.length(); buffer = new char[len + 1]; for (int i = 0; i < len; i++) buffer[i] = right[i]; buffer[len] = '\0'; }
  • Sole parameter is a reference to an object of the same class
  • Parameter really should be const

Quality Tip

When to Use the System-Defined Copy Constructor

  • Copy constructor is supplied
  • Default behavior is to perform a member-wise (or shallow) copy
  • Typically, an explicit copy constructor is needed when an object stores references to external resources (such as heap memory)

Field Initializer Lists

Member data may be assigned in a field initializer list

  • Syntax only valid for constructors
  • Used to avoid modifying an attribute twice (default constructor, then in the body of the outer constructor) Employee::Employee(String employee_name, double initial_salary) : name(employee_name), salary(initial_salary) { }
  • Is the only way to invoke a non-default constructor of a base class TeachingAssistant::TeachingAssistant(String student_name) // Teaching assistants all get same starting salary : Employee(student_name, 5000) { }

18.3.4 Field Initializer Lists

Initializer lists must be used for:

  • Data fields declared as non-static const
  • References
  • To invoke a base class' constructor (other than default)
  • Assigning a size to a vector member: PartDescription::PartDescription(String part_name, int inventory_number) : name(part_name), part_number(inventory_number), subcomponents(3) {}

Assignment Operators

  • Tasks performed are similar to copy constructor
  • Must do cleanup first
  • Check for auto-assignment

Assignment Operators

  • Example: String class: String& String::operator=(const String& right) { if (this != &right) { delete[] buffer; // Get rid of old buffer len = right.length(); buffer = new char[len + 1]; for (int i = 0; i < len; i++) buffer[i] = right[i]; buffer[len] = '\0'; } return *this; }

Destructors

  • Perform necessary housekeeping tasks
  • Named ~ClassName, no arguments, no return type
  • Implicitly called on:
    • Local variables, at end of block
    • Parameters, at end of function
    • Temporary variables, at end of statement
    • Heap values, when delete is called
    • Static values, when main terminates
    • Member data, when object is deallocated
    • Base class object, when derived class object is deallocated
    • Local variables, when an exception is thrown and execution leaves the block

Syntax Destructor Definition

ClassName::~ClassName() { statements }

Example:

String::~String() { delete[] buffer; }

Purpose:

Perform any housekeeping tasks that should be performed before an object is deleted.

Common Error

Confusing Destruction and Deletion

Related, but distinct concepts

  • Deletion occurs when a pointer is passed to delete
  • Destruction occurs when a class' destructor is invoked

Pointers are deleted, and objects are destroyed

  • delete will invoke an object's destructor String* p = new String("Alice"); ... delete p; // p->~String() automatically invoked here

Quality Tip

Include Virtual Destructors

  • Any class that has a virtual method should define a virtual destructor (even if it does nothing)
  • E.g.: Consider the following declarations class Person { ... // No destructor private: int id; }; class Employee : public Person { ... private: string name; };

Quality Tip

Include Virtual Destructors

  • Now imagine that you create and delete a polymorphic variable, as follows: Person* a = new Employee(); delete a;
  • Employee's destructor is never called
  • name string is never destroyed
  • Memory leak

Quality Tip

If Destructor, then Copy Constructor and Assignment

The "Big Three"

  1. Copy constructor
  2. Assignment operator
  3. Destructor
  • All have default behavior, as does the default constructor
  • Behavior sometimes needs to be extended
  • Reason to define one → reason to define all three
  • Build all 3 on lower-level helper methods, so they behave the same

Quality Tip (cont.)

Consider our String class

  • Imagine we supplied a destructor, but no copy constructor String a = "Fred"; // Internal buffer is allocated here { String b = a; // Error.memberwise copy produces shared buffer } // Destructor b.~String() invoked, buffer is deleted cout << a << "\n"; // Sure to cause error, because buffer is now deleted

Quality Tip (cont.)

  • Notice that the big three are built on two lower-level operations: X::X(const X& right) { copy(right); } X& X::operator=(const X& right) { if (this != &right) { free(); copy(right); } return *this; } X::~X() { free(); }

The Class auto_ptr

  • "Smart pointers"
  • Declared in <memory>
  • Good when the lifetime of the dynamic value is tied to another object
  • Dynamic memory auto_ptr references is automatically recovered int f() { auto_ptr<Employee> p = new Employee(...); ... } // Destructor of p will automatically delete Employee object
  • Very useful with exceptions

Advanced Topic

Overloading Memory Management Operators

  • It is possible to overload new and delete
  • E.g., supply allocators for the Node class
  • Deleted values would be stored in a global linked list
  • A request to new would first check this list class Node { public: Node(string s); void* operator new(size_t bytes); void operator delete(void* value); private: string data; Node* previous; Node* next; };

Advanced Topic (cont.)

Overloading Memory Management Operators

Node* free_store_list = NULL; // Global list, initially empty void* Node::operator new(size_t types) { if (free_store_list != NULL) { result = free_store_list; free_store_list = free_store_list->next; } else result = ::new Node; return result; } void Node::operator delete(void* value) { Node* p = static_cast<Node*>(value); p->next = free_store_list; free_store_list = p; }

Advanced Topic (cont.)

Overloading Memory Management Operators

  • Arguments and return types must be as shown
  • Note the global scoping (::) of new and delete
  • Note that free_store_list will, in turn, need to be cleaned up

  • Simple rule: objects that allocate memory should clean it up
  • Not usually that simple. Consider sharing:
  • Two options:
    1. Designate one as owner (not always possible)
    2. Use reference counting

  • When data object is shared, reference count is incremented:
  • When a references is destroyed (or reassigned), ref count is decremented:
  • When the reference count is zero, the shared resource is deleted

Reference Counting

Consider our String class: change the semantics for the assignment of strings so that two strings would share a common internal data buffer:

  • StringReference — nested class holding reference count and the buffer
  • References are managed by reassign()
    • Takes ptr to a StringReference
    • If not NULL, reference to argument value is incremented
    • Reference to current value is decremented
    • If current reference is 0, object is deleted
    • Current object is set to refer to argument value

Common Error

Reference Counting Fails in the Presence of Cycles

  • Reference counting has some drawbacks:
    • Objects are slightly larger, increasing copying overhead (negligible)
    • It does not work for more complex data structures
