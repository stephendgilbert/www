Functions as Black Boxes

  • Some functions take more than one argument, or none
    • pow(x, y) computes xy
  • Many functions have output or return values:
    • sqrt() returns the answer: y = sqrt(b * b - 4 * a * c)
    • One return value, at most
  • Parameters (variables that hold arguments) and return values are typed

Function Comments

  • State the purpose of the function
  • Describe the preconditions and the postconditions:
    precondition
    Anything that must be true before the function is called
    Describe the type and legal values for the arguments
    Any other requirements, such as global variables that must exist (avoid this!)
    postcondition
    What will be true after the function executes (given that the preconditions were met)
    Describe the return value
    Any other side-effects

Function Comments

  • For humans to read; compiler never sees them
  • Manual for anybody who wants to use the function (client)
  • No standard layout
  • Think description/inputs/output/side-effects, or
  • description/preconditions/postconditions
  • We'll use javadoc-style tags in this book
  • Tools exist that will generate documentation (HTML, RTF, etc.) from source

Function Comments

Function Comments

  • Good function and parameter names should still be chosen
  • Black-box description; not the code
  • Write the comments before writing the function
    • You understand the specs
    • The function follows the specs
    • (Rather than the specs following the code)

Return Values

  • return statement specifies the result of a function
  • When the return statement is processed, the function exits immediately
  • This is convenient for handling exceptional cases: double future_value(double initial_balance, double p, int n) { if (n < 0) return 0; if (p < 0) return 0; double b = initial_balance * pow(1 + p / 100, n); return b; }

Return Values

  • Return values can be any expression (of the right type) ) double future_value(double init_bal, double p, int n) { return initial_balance * pow(1 + p / 100, n); }
  • Every branch must return a value double future_value(double init_bal, double p, int n) { if (p >= 0) return initial_balance * pow(1 + p / 100, n); // Error }
  • Last statement should always be a return statement

Return Values — predicates

Predicate functions

  • Return a truth value (bool)
  • Have no side effects
  • E.g., function to check if input is even: bool is_even(int n) { return n % 2 == 0; }
  • Can be used inside a test: if (is_even(input)) …
  • You've already seen this: if (cin.fail()) …

Parameters

  • Values passed into a function are called arguments, or parameter values
  • The variable in a function that (initially) holds an input value (argument) is called a formal parameter, parameter variable, or just a parameter

When a function is called:

  • All expressions (arguments) are evaluated before the function is called
  • Parameters are initialized with these values (in order)

Side Effects

Externally observable effect (other than the return value) caused by a function call. Something in the state of the machine that changed since the function was called. E.g.:

  • Changing a global variable
  • Printing to the screen
  • Reading from the keyboard
  • Ending the program
  • Changing reference parameters (soon)
  • Erasing the hard-drive
  • Moving an arm on an assembly line

Side Effects

  • Note, future_value doesn't print the answer (or anything) to the screen. It just does its job
  • Printing error messages inside a function is extremely reprehensible (unless that is the function's job)
  • These functions wouldn't work where cout doesn't exist
  • Limits reusability
  • Group I/O activity together
    • Abstract out (to functions) for larger programs

Side Effects

  • Endeavor to write functions without side effects
  • Side effects can't entirely be avoided
  • Some functions are implemented solely for their side effects
  • But, functions without side-effects make programming much simpler
    • A function without side effects always behaves the same
    • Much easier to debug
  • Most functions have no side effects

Procedures

  • A function without a return value is called a procedure print_time(now);
  • This distinction is blurred in C++
  • They're all functions in C++
  • The missing return value is indicated by the keyword void void print_time(Time t) { cout << t.get_hours() << ":" << setw(2) << setfill('0') << t.get_minutes() << ":" << setw(2) << t.get_seconds() << setfill(' '); }
  • Since a procedure does not have a return value, it must have a side effect

Procedures

  • A procedure might return a status
  • E.g., print_paycheck() might return a bool if it printed successfully (paper in the tray, no jam, enough ink, etc.)
  • This return value is not the purpose of the procedure

Reference Parameters

  • Arguments (parameter values) are copied into the function, by default
    • Called pass by value
    • Modifying the parameter in the function does not change the argument in the caller
  • For changes to a parameter to persist, we use a reference parameter
    • Does not create a new variable
    • A reference to the argument is actually passed in
    • Modifying the parameter modifies the argument in the caller
    • Argument must be an l-value (something that can be modified)

Reference Parameters

  • Consider a function that raises an Employee's salary, by a given percentage: Employee harry; ... raise_salary(harry, 5); // Now Harry earns 5 percent more
  • Reference parameters are indicated using an & after the parameter's type: void raise_salary(Employee& e, double by) { double new_salary = e.get_salary() * ( 1 + by / 100); e.set_salary(new_salary); }

Reference Parameters

  • The parameter by is a value parameter; it's initialized with a copy of the value supplied by the caller
  • e is a reference parameter; it actually refers to the variable used in the call

Figure 7 Reference and Value Parameters

Reference Parameters

  • A reference parameter refers to different variables in different calls
  • Here e refers first to harry, then to charley raise_salary(harry, 5 + bonus); raise_salary(charley, 1.5);
  • by should not be a reference parameter
    • It's not modified in the function
    • Reference parameters must be bound to a variable, so raise_salary(harry, 5 + bonus) would be illegal

Advanced Topic

Constant References

  • Passing large objects into functions is inefficient
    • Takes time to copy
    • Takes more space
  • We can pass by reference, avoid the overhead
  • If there's no need for the function to modify the object, make it a const reference void print_employee(const Employee& e) { cout << "Name: " << e.get_name() << " Salary: " << e.get_salary() << "\n"; }

Advanced Topic

Constant References (cont.)

  • Constant reference parameters can not be modified in a function
  • Only const methods may be called on const objects
  • This decision should be made when the function is implemented
  • Can be gleaned from the specs
  • More on this subject (const-correctness) later

Variable Scope and Global Variables

  • Same variable name can be used in different functions
  • Consider: double future_value(double initial_balance, double p, int n) { double r = initial_balance * pow(1 + p / 100, n); return r; } int main() { cout << "Please enter the interest rate in percent: "; double r; cin >> r; double balance = future_value(10000, r, 10); cout << "After 10 years, the balance is " << balance << "\n"; return 0; }

Variable Scope and Global Variables

  • These variables r are independent of each other
  • Where a particular variable is visible is its scope
  • Variables defined inside a function:
    • Called local variables
    • Their scope extends from the definition to the end of the enclosing block
  • Variables defined outside functions are call global
    • Visible to all functions that are defined after it

Preconditions

  • Anything that a function assumes upon invocation
    • Argument to sqrt is non-negative
    • Argument to cos is in radians
  • Preconditions should be commented
    • If a precondition is not met, function can do what it wants

Preconditions

  • Sometimes a function can check a precondition
    • Is the argument non-negative?
  • Sometimes not
    • Is this character array null-terminated?
    • Is that float measured in radians?

Preconditions

If a function can determine inappropriate inputs, what should it do?

  • Return default value
    • E.g., digit_name returns an empty string when called with an unexpected value
    • This is a bad idea. Even if a good sentinel (invalid output) exists, we rely on the caller to check
  • Terminate
    • Ugly, and not informative
  • Throw an exception
    • Many functions in the cmath library throw an exception if given an illegal value
    • Requires deeper study
  • Use the assert() mechanism

Preconditions — assert()

  • Only useful in debugging
  • Given a condition that tests false, prints an error message, then dumps the program
  • Otherwise, does nothing assert(x >= 0);
  • Behavior can be modified in released code, so that assert statements are ignored entirely

Syntax : Assertion

assert(expression)

Example: 

assert(x >= 0);

Purpose:

If the expression is true, do nothing. If it is false, terminate the program, displaying an error message with the file name, line number, and expression.

Unit Testing

Testing can only show the presence of bugs, not their absence — Edsger Dijkstra

block-box testing
Test functionality of a program (or function) without consideration of its internal structure
Can (should) be written from the specs, before the code is written
Not possible (usually) to test all inputs
white-box testing
Look at internal structure
Exercise each code path
Test border conditions

Unit Testing

  • Unit testing — process of testing each function in isolation
  • Write a test harness that calls a function with various inputs int main() { assert(int_name(19) == "nineteen"); assert(int_name(29) == "twenty nine"); assert(int_name(1093) == "one thousand ninety three"); assert(int_name(30000) == "thirty thousand"); }

Unit Testing

  • Unit test frameworks exist for larger programs
  • Easier to organize unit tests
  • Automates testing
  • More sophisticated

Unit Testing — selecting cases

  • Select "typical" inputs
  • Include boundary cases
    • Legitimate inputs
    • Examples for int_name: 1, 999999
  • Cover each branch (test coverage)
    • Each if/else branch, each loop
    • Examples for int_name: 1, 10, 19, 20, 27, 99, 100, 101, 115, 121, 999, 1000, 1003, 1021, 1221, etc.

Unit Testing

  • Write test cases before implementing a function
  • It is common to add test cases for each but that is discovered and fixed
    • Leave in the test suite
    • A bug often reappears after being "fixed" (cycling)
  • regression testing — testing against a set of past failures

Using a Debugger — 3 useful commands

Three commands that will get you far

break — "run until this line"
Execution pauses at (before) this line or function
Displays next line to be executed
Allows user to look at values of variables at this point
step — "execute current line, stop at next"
You can step into a function call, watch the function execute, or
You can execute a function call as a single line, going to the next line
print — "inspect variable"
While halted, debugger allows you to see the value of any variable (in scope)
In fact, allows you to evaluate any valid expression

Stepping Through a Program

There are (at least) 2 step commands

step over (next)
Executes an entire line of code
Stops at the next line
step into (step)
If the line contains a function call, will step into it, stopping at the first line in the called function
Some debuggers have a step up command that will finish that function, take you back to the next line in the caller
Rarely useful to step into system functions

Stepping Through a Program — breakpoints

  • Programs in the debugger can be started in by stepping into it (starting w/the first line in main()), OR
  • Setting a breakpoint
    • On a line number, a function, or a condition
  • Program runs normally until a breakpoint is encountered
  • Then you may step, continue to the next breakpoint

    • Did the error occur already? Then error is between beginning and here
    • Else, error is between here and the end
    • Cut the interesting piece of code in half; repeat

Debugging Strategies

Understand What the Program Should Do

  • Debugger tells you what a program is doing, not what it should do
  • E.g., before going through a loop, know when it should end
  • Know what a function should return
  • Before you look at a variable, know what it should be

  • Will probably cause trouble elsewhere
